## Context

YMOS V3 的个股投资流程覆盖了从关注→调研→买入→持有→卖出的完整链路，通过 P 系列提示词（P1-P16）和 5 条策略路由实现分析编排。但在三个环节存在结构性缺失：

- **买入路由缺少横向视角**：Route A/B 只分析目标标的本身，不与 Watchlist 中其他候选做比较
- **卖出后的经验无法回流**：P11 Autopsy 存在但不是强制环节，交易经验散落在报告中，无法反哺系统
- **仓位管理停留在粗粒度**：投资偏好定义了 PVE/PVP/现金大类比例，但缺少单笔仓位、分批规则、集中度控制

系统基于 Markdown SOP + 提示词软路由架构，新能力需要遵循同样的模式：新增或扩展 SOP 步骤、增加 P 系列提示词、扩展投资偏好文件。

## Goals / Non-Goals

**Goals:**

- 买入/加仓时自动识别候选标的并输出标准化对比排名
- 卖出/清仓后强制执行结构化复盘，结论写回知识库和投资偏好
- 基于用户风险参数自动计算单笔仓位、分批建仓规则、集中度预警
- 所有变更遵循 Markdown-First 原则，不引入新的数据格式或工具依赖

**Non-Goals:**

- 不做技术分析 / K 线 / 量价指标（超出系统定位）
- 不做自动交易信号或程序化执行
- 不改变现有 P1-P16 的核心逻辑，只扩展调用时机和上下文
- 不引入数据库或非 Markdown 存储

## Decisions

### D1: 横向对比嵌入买入路由，而非独立 SOP

**选择**：在 `SOP_策略分析.md` 的 Route A/B 中增加「Step 3.5 候选标的横向对比」步骤

**理由**：横向对比是买入决策的子环节，独立成 SOP 会增加用户认知负担。嵌入路由后，用户仍说 `我想买 TSLA`，系统自动在后台完成对比。

**替代方案**：新建独立暗号 `对比一下 TSLA 和 NVDA` → 额外的入口和路由维护成本，且容易忘记执行。

### D2: P11 升级为结构化复盘模板，卖出路由强制调用

**选择**：升级 `Brain/references/p11-autopsy.md` 为结构化模板（买入逻辑回溯 / P1P4 偏差分析 / 阈值校准），在 Route D 完成后作为必经步骤执行

**理由**：当前 P11 是可选提醒，实际执行率低。强制化 + 结构化模板确保每次卖出都有经验回流。

**替代方案**：创建独立的 P17 复盘模块 → 增加 P 系列复杂度，P11 本身就是复盘定位，升级更合理。

### D3: 新增 P17 仓位计算器提示词，嵌入所有涉及仓位的路由

**选择**：新增 `Brain/references/p17-position-sizing.md`，在 Route A（买入）、Route B（加仓）、Route E（再平衡）中按需调用

**理由**：仓位计算是跨路由的通用能力，提取为独立提示词可复用。计算参数从 `当前关注方向与投资偏好.md` 读取。

**替代方案**：在 P7 Portfolio Check 中扩展仓位计算 → P7 定位是组合级再平衡，混入单笔计算会模糊职责边界。

### D4: 集中度预警通过投资偏好中的阈值规则实现

**选择**：在 `当前关注方向与投资偏好.md` 增加「仓位管理参数」区块，定义单笔上限、板块集中度阈值、关联度矩阵。P17 读取这些参数进行校验。

**理由**：集中度规则因人而异，放在投资偏好中符合"你的系统你做主"理念，且 P12 纪律审查也可以引用这些参数。

## Risks / Trade-offs

- **[横向对比增加分析耗时]** → 只对 Watchlist 中同类型（同 PVE/PVP 分类）的标的做对比，不跨类比较；对比维度限制在 P2 + P9 的标准化子集
- **[强制复盘可能被用户跳过]** → 系统层面标记为必经步骤，但在对话中提示用户可以简略执行（至少回答模板中的核心问题）
- **[仓位计算参数需要用户填写]** → 提供默认值（单笔上限 10%、单板块上限 30%），用户不填写时使用默认规则
- **[P 系列提示词数量增长]** → P17 是必要补充，但需要控制后续增长。新增 P 系列需有明确的不可替代性论证
