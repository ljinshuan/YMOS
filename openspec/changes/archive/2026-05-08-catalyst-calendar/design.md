## Context

YMOS 的 ymos-radar 可以检测当前发生的事件（通过 RSS、富途数据源等），但这些信息是反应式的。用户无法提前规划：
- 哪些公司即将发布财报？
- 下周有哪些行业会议/产品发布会？
- 什么时候有重要的宏观数据发布？

Financial Services 的 Catalyst Calendar 提供了前瞻性事件管理功能。但 YMOS 需要适配：
1. 多市场支持（A股/港股/美股）
2. 与现有持仓/观察标的联动
3. 轻量级实现，不过度依赖外部数据

## Goals / Non-Goals

**Goals:**
- 建立催化剂日历系统，收集和展示即将到来的关键事件
- 提供每周预览功能，帮助用户规划交易
- 评估事件对持仓的影响，提供风险预警
- 支持事件与论点/持仓的关联
- 支持日历导出（Excel 格式）

**Non-Goals:**
- 不实时推送事件通知（由用户主动查看）
- 不自动根据事件执行交易
- 不维护历史事件数据库（关注未来事件）

## Decisions

### 1. 日历存储位置

**决策：** 全局日历 + 个股关联的双层结构

**结构：**
- `data/reports/catalyst-calendar/YYYY-MM/催化剂日历.md` — 全局日历
- 个股文件夹可选关联催化剂（通过引用全局日历）

**理由：**
- 全局日历统一管理，避免重复
- 个股关联实现个性化视图
- 支持跨标的的事件分析（如行业会议影响多个标的）

**备选方案：** 仅在个股文件夹中存储
- 不利：无法看到跨标的宏观事件，重复率高

### 2. 事件类型分类

**决策：** 四大分类

**分类：**
1. **Earnings & Financial** — 财报、股东会、分析师日
2. **Corporate** — 产品发布、FDA 批准、M&A 里程碑、管理层变动
3. **Industry** — 行业会议、贸易展、监管裁决
4. **Macro** — FOMC、CPI、GDP、央行决策、地缘政治

**理由：**
- 覆盖主要价格驱动因素
- 与 Financial Services 分类一致
- 便于筛选和优先级排序

### 3. 影响评估

**决策：** 三级影响 + 仓位影响

**格式：**
```markdown
- 影响等级: High/Medium/Low
- 仓位影响: [持仓列表]
- 预期: [简要描述可能的影响]
```

**理由：**
- 三级足够区分重要性
- 明确关联持仓，提供可操作建议
- 保留预期描述，供用户参考

### 4. 每周预览格式

**决策：** 结构化 Markdown 格式

**格式：**
```markdown
# 催化剂周报 — YYYY-MM-DD 至 YYYY-MM-DD

## 本周关键事件
1. **[日期] [公司/事件]** — [影响等级]
   - 为什么重要: ...
   - 仓位影响: ...

## 下周前瞻
- [日期]: [事件] — [潜在影响]

## 风险提示
- [需要特别关注的事件]
```

**理由：**
- 简洁易读，适合快速浏览
- 结构清晰，便于机器解析
- 与 YMOS 报告风格一致

### 5. 数据源策略

**决策：** 多源采集 + 手动补充

**数据源：**
- 财报日期：Futu API / Yahoo Finance
- 宏观事件：公开日历（手动维护）
- 企业事件：用户手动添加 + 可选 RSS 抓取
- 行业事件：手动维护 + 可选爬虫

**理由：**
- 财报日期可通过 API 自动获取
- 宏观/行业事件相对固定，手动维护更可靠
- 保留手动补充渠道，覆盖 API 无法获取的数据

### 6. 与现有 Skill 集成

**决策：** 创建独立的 `ymos-catalyst-calendar` skill

**集成点：**
- `ymos-radar`: 事件触发后，询问是否添加到日历
- `ymos-strategy`: 策略分析前，读取即将到来的催化剂
- `ymos-thesis-tracker`: 催化剂可关联到论点支柱

**理由：**
- 独立 skill 符合 YMOS 架构
- 集成点明确，耦合度低
- 便于未来扩展（如添加自动提醒）

### 7. Excel 导出

**决策：** 使用 Python openpyxl 生成标准 Excel 格式

**Excel 格式：**
- Sheet 1: 日历视图（按日期排序）
- Sheet 2: 按标的分组
- Sheet 3: 按影响等级排序

**理由：**
- Excel 是投资机构的通用格式
- 多 Sheet 支持多种视图
- 便于用户导入到自己的日历系统

## Risks / Trade-offs

### Risk 1: 数据准确性问题

**风险：** 财报日期可能被公司临时调整

**缓解：**
- 提示用户在临近日期时再次确认
- 在日历中标注"预估" vs. "确认"
- 考虑添加历史准确率统计

### Risk 2: 手动维护负担

**风险：** 宏观/行业事件需要手动维护，用户可能遗忘

**缓解：**
- 提供标准模板，降低输入成本
- 预填充常见事件（如 FOMC 会议）
- 考虑未来集成第三方日历 API

### Risk 3: 信息过载

**风险：** 日历事件过多，用户难以聚焦

**缓解：**
- 按影响等级优先排序
- 每周预览只列出关键事件
- 支持筛选（按标的、类型、影响等级）

## Migration Plan

**部署步骤：**
1. 创建 `ymos-catalyst-calendar` skill 目录和文件
2. 创建日历模板和每周预览模板
3. 更新 `ymos-radar` skill，添加日历集成
4. 更新 `ymos-strategy` skill，添加催化剂读取
5. 创建 CLI 工具 `ymos catalyst-calendar` 用于数据抓取
6. 更新 CLAUDE.md 和路由文档

**回滚策略：**
- 所有变更都是新增或修改，可通过 git 回滚
- 不影响现有数据结构

## Open Questions

1. **历史事件处理：** 过去的事件如何处理？
   - 建议：从日历中移除，保留在每周报告的"本周回顾"中

2. **事件重复：** 同一事件关联多个标的时，如何避免重复？
   - 建议：在全局日历中存储一次，通过"仓位影响"字段关联多个标的

3. **时区处理：** A股/港股/美股时区不同，如何统一显示？
   - 建议：以用户本地时区为主，在事件详情中标注原始时区

4. **自动提醒：** 未来是否需要添加自动提醒功能？
   - 建议：V2 功能，当前保持用户主动查看模式
