## Context

YMOS 当前通过 P1（Genesis 基石建档）、P4（Radar 关注点雷达）、P2（Phase Check 阶段判断）构建个股知识库。这些 prompt 产生静态分析结果，但缺乏持续追踪机制。用户在持有/观察股票期间会不断接收到新信息（财报、新闻、行业动态），但系统无法：
1. 记录这些信息对原有论点的影响
2. 量化论点置信度的变化趋势
3. 提醒用户关键催化剂的临近

Financial Services 的 Thesis Tracker 提供了一个很好的参考：论点记分卡、更新日志、催化剂日历。但 YMOS 需要适配其现有的 P 系列框架和 Markdown 数据层。

## Goals / Non-Goals

**Goals:**
- 建立投资论点追踪机制，让用户可以持续验证和更新投资论点
- 提供论点记分卡，可视化关键支柱的状态和趋势
- 集成更新日志，记录每个数据点对论点的影响
- 关联催化剂日历，预警即将到来可能验证/证伪论点的事件
- 与现有 P 系列框架无缝集成

**Non-Goals:**
- 不自动生成买入/卖出决策（由 ymos-strategy 负责）
- 不实时抓取新闻/财报数据（由 ymos-radar 负责）
- 不改变现有 P1/P4/P2 的核心逻辑

## Decisions

### 1. 论点追踪文件结构

**决策：** 在个股文件夹内创建独立的 `投资论点追踪.md` 文件

**理由：**
- 与现有的 `个股基础知识库.md` 分离，保持单一职责
- 支持多次更新而不污染基础档案
- 便于导出为独立报告

**备选方案：** 将追踪内容追加到 `个股基础知识库.md`
- 不利：文件会变得很长，难以维护

### 2. 论点置信度量化

**决策：** 使用三级置信度 + 趋势方向

**格式：**
```
置信度: [High/Medium/Low] (↑/↓/→)
```

**理由：**
- 三级足够区分强/中/弱论点
- 趋势方向显示变化，支持历史回溯
- 简单直观，无需复杂计算

**备选方案：** 数值评分（1-10）
- 不利：缺乏语义，用户难以理解

### 3. 记分卡格式

**决策：** 使用 Markdown 表格

**格式：**
```markdown
| 支柱 | 原始预期 | 当前状态 | 趋势 | 最近验证日期 |
|------|----------|----------|------|--------------|
| 支柱1 | ... | On Track/Behind/Ahead | ↑/↓/→ | YYYY-MM-DD |
```

**理由：**
- Markdown 表格易读易维护
- 与 YMOS 现有格式一致
- 便于渲染为可视化图表（未来扩展）

### 4. 更新日志结构

**决策：** 使用时间倒序的日志格式

**格式：**
```markdown
## 更新日志

### 2025-05-08
- **事件**: Q1 财报发布
- **数据点**: 收入 120M vs 预期 115M (+4.3%)
- **影响支柱**: 收入增长
- **论点影响**: Strengthen
- **操作建议**: No change
- **置信度变化**: Medium → High (↑)
```

**理由：**
- 时间倒序便于查看最新动态
- 结构化字段便于解析和统计
- 明确记录因果关系

### 5. 催化剂集成

**决策：** 催化剂日历嵌入论点追踪文件，不独立创建文件

**理由：**
- 催化剂与论点紧密相关，不应分离
- 减少文件数量，降低维护成本
- 可在 catalyst-calendar change 中复用格式

### 6. Skill 集成方式

**决策：** 创建独立的 `ymos-thesis-tracker` skill，通过依赖被其他 skill 调用

**集成点：**
- `ymos-research`: 在 P1/P4 完成后，询问是否初始化论点追踪
- `ymos-strategy`: 在策略分析前，读取当前论点置信度作为输入
- `ymos-radar`: 在事件触发后，提示是否更新论点追踪

**理由：**
- 独立 skill 符合 YMOS 的 skill 分离原则
- 依赖关系通过 `depends_on` 元数据声明
- 保持各 skill 的可测试性

## Risks / Trade-offs

### Risk 1: 论点追踪与现有知识库重复

**风险：** `投资论点追踪.md` 与 `个股基础知识库.md` 可能有内容重叠

**缓解：**
- 明确分工：基础知识库是静态事实，论点追踪是动态观点
- 在 SOP 中规定：基础知识库不包含观点判断
- 在论点追踪中引用基础知识库作为事实来源

### Risk 2: 更新频率不确定

**风险：** 用户可能忘记更新论点追踪，导致数据过时

**缓解：**
- ymos-radar 触发事件时，主动提示更新
- 在 ymos-strategy 中检查论点追踪的最近更新日期
- 考虑在下次迭代中添加自动提示机制

### Risk 3: 格式标准化挑战

**风险：** 用户可能手动编辑导致格式不一致

**缓解：**
- 提供标准模板
- skill 中包含格式校验
- 在文档中明确说明编辑规则

## Migration Plan

**部署步骤：**
1. 创建 `ymos-thesis-tracker` skill 目录和文件
2. 创建论点追踪模板文件
3. 更新 `ymos-research` skill，添加论点初始化选项
4. 更新 `ymos-strategy` skill，添加论点置信度读取
5. 更新 `ymos-radar` skill，添加论点更新提示
6. 更新 CLAUDE.md，添加新 skill 的说明

**回滚策略：**
- 每个步骤都是新增文件或修改文件，可通过 git 回滚
- 不影响现有数据结构（新增的论点追踪文件是可选的）

## Open Questions

1. **历史数据迁移：** 是否需要为现有持仓/观察标的创建历史论点追踪？
   - 建议：按需创建，不强制迁移

2. **多持仓场景：** 用户有多个持仓时，如何批量更新论点追踪？
   - 建议：未来迭代考虑批量操作接口

3. **论点归档：** 当股票清仓或移除观察时，论点追踪如何处理？
   - 建议：保留在个股文件夹中，作为历史记录

4. **可视化：** 是否需要将记分卡渲染为图表？
   - 建议：V2 功能，当前保持 Markdown 表格格式
